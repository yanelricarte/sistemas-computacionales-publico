1. Punto de partida

Veníamos trabajando con un login básico: una página con formulario de usuario y contraseña que valida los datos y muestra un mensaje. Funciona, pero todavía no es un sistema:

  • Una vez que validás, el sistema no recuerda quién entró.
  • Cualquiera que escriba la URL de una página interna entra sin pasar por el login.
  • El estilo visual es desparejo entre archivos.
  • No hay navegación clara entre páginas del proyecto.

Estos son los cinco temas que vamos a incorporar para que ese login deje de ser una página suelta y pase a ser un mini-sistema web funcional:

  • CSS común en un solo archivo que aplica a todas las páginas.
  • Redirecciones server-side para mover al usuario entre páginas después de procesar algo.
  • Sesiones para que el sistema recuerde quién está logueado al pasar de una página a otra.
  • Protección de páginas internas para que sólo los usuarios logueados puedan entrar.
  • Navegación por URL con GET (una sola página que muestra contenido distinto según un parámetro).

Al final de la clase vas a tener un mini-sistema web. Sigue chiquito, sigue sin base de datos, sigue con usuarios hardcodeados. Pero ya tiene forma de sistema.

Una sola carpeta del proyecto

Antes de tocar código, una cosa de hábito profesional que vale para todo el año:

Todo lo de esta clase vive dentro de una carpeta única, por ejemplo clase02/. Adentro están los archivos PHP, una subcarpeta includes/, una subcarpeta css/. Es la carpeta entera la que es tu proyecto, no los archivos por separado.

Abrí el editor "en la carpeta", no archivo por archivo

Cuando abrís VS Code (u otro editor) hacé Archivo → Abrir carpeta... y elegí clase02/. No abras los archivos sueltos uno por uno.

¿Por qué importa? Porque tus archivos se referencian entre sí con rutas relativas (includes/cabecera.php, css/estilos.css). Si abrís un archivo solo, el editor no entiende el contexto: las rutas no autocompletan, los includes no se navegan, y cuando algo no funciona no tenés cómo ver el árbol completo del proyecto.

Abrir la carpeta es lo que hace todo desarrollador profesional. Acostumbrate desde ahora.

Cómo se ve "abrir la carpeta" en VS Code
  1. VS Code abierto, sin nada cargado.
  2. Menú Archivo → Abrir carpeta... (o Ctrl+K Ctrl+O).
  3. Navegás hasta clase02/ y la seleccionás.
  4. El panel de la izquierda (Explorador) ahora muestra todo el árbol del proyecto: archivos PHP en la raíz, includes/ y css/ como subcarpetas. Podés clickear cualquier archivo y navegar entre ellos sin perderte.

Estructura recomendada del proyecto

Así es como va a quedar tu carpeta clase02/ al final de la clase. Esto es lo que vas a armar paso a paso; no esperés tenerlo todo de entrada:

clase02/
├── index.php              ← formulario de login
├── procesar-login.php     ← valida usuario + clave, abre sesión, redirige
├── bienvenido.php         ← página interna protegida por sesión
├── info.php               ← una página, contenido distinto según ?seccion=...
├── logout.php             ← cierra la sesión y vuelve al login
│
├── includes/
│   ├── cabecera.php       ← <head>, <link CSS>, navbar
│   └── pie.php            ← cierre de <body> y <html>
│
└── css/
    └── estilos.css        ← todos los estilos del sitio en un solo archivo

Cada archivo tiene una responsabilidad clara. Cuando algo no funciona, ya sabés dónde buscar: si el estilo está raro, mirás estilos.css; si la navbar no aparece, mirás cabecera.php; si después de loguearte no entrás, mirás procesar-login.php.

No es la única forma de organizarlo
Esta estructura es una convención razonable para proyectos chicos de PHP, pero no es la única. En proyectos más grandes hay frameworks (Laravel, Symfony) que imponen estructuras propias. Por ahora, esta forma te alcanza para todo lo del año hasta MySQL.

Cómo funciona la web por debajo: HTTP en 2 minutos

Antes de meternos en CSS, sesiones y redirecciones, conviene tener claro cómo viaja la información entre tu navegador y el servidor. El protocolo se llama HTTP (HyperText Transfer Protocol). Es el lenguaje con el que se hablan navegador y servidor.

Petición y respuesta

Cada vez que escribís una URL o clickeás un link, el navegador manda una petición (request) al servidor y el servidor devuelve una respuesta (response). Eso es todo. Cada interacción con la web es ese ciclo.

Navegador                              Servidor
   |                                       |
   |---- PETICIÓN (GET / POST) ----------->|
   |     "Dame esta URL"                   |
   |     o "Acá van estos datos"           |
   |                                       |  procesa
   |<---- RESPUESTA --------------------- |
   |     status: 200 / 404 / 302 ...       |
   |     headers: Content-Type, etc.       |
   |     body: HTML, CSS, JSON, ...        |
   |                                       |

Métodos HTTP (verbos)

La petición tiene un método que indica la intención. Los dos que usamos este año:

MétodoPara quéDónde van los datos
GETPedir información (mostrar una página, buscar, filtrar)En la URL, después del ?: info.php?seccion=ayuda
POSTEnviar datos al servidor (formularios, login, alta de algo)En el cuerpo de la petición, ocultos en la URL

Existen otros (PUT, DELETE, PATCH...) que aparecen cuando hablemos de APIs, pero los dos de arriba son el 95% del tráfico web normal.

Códigos de respuesta (status codes)

El servidor devuelve un número de 3 dígitos que dice cómo fue la cosa. Los que vas a ver más:

CódigoSignificaCuándo
200OKTodo bien, acá está tu respuesta.
302RedirecciónCuando PHP hace header('Location: ...'). Le dice al navegador "andá a otra URL".
404Not FoundEl archivo o ruta no existe. Típico cuando tipeás mal una URL o falta un archivo.
500Internal Server ErrorPHP rompió. Hay un error en tu código. Mirá el log de errores.

Headers HTTP

Tanto la petición como la respuesta llevan, antes del contenido, una lista de headers (encabezados) con información sobre el pedido. Por ejemplo: qué tipo de contenido viene (Content-Type: text/html), qué idioma acepta el navegador, si hay cookies, etc.

El header que más vamos a tocar este año se llama Location, y es el que usamos en las redirecciones server-side (sección 3). Cuando PHP manda un Location:, el navegador entiende que tiene que ir a otra URL.

Lo más importante: HTTP es "sin memoria"

Cada petición es independiente de las anteriores. El servidor, por sí solo, no sabe que vos ya pediste una página antes; cada vez que pedís algo, sos un desconocido nuevo para él.

Eso es un problema para cosas como un login: si me logueo y después pido la página de bienvenida, ¿cómo sabe el servidor que soy yo? La solución se llama sesiones (sección 5 — paciencia, ya llegamos).

Cómo ver todo esto en el navegador

Abrí cualquier página, apretá F12 (DevTools) y andá a la pestaña Network / Red. Refrescá la página: vas a ver cada petición que hace el navegador (HTML, CSS, JS, imágenes...) con su método, su status code y sus headers. Hacé click en cualquier fila para ver los detalles. Es una herramienta indispensable cuando algo no funciona.

2. CSS en archivo aparte

Hasta esta clase el CSS no estuvo en foco: cada uno tomó decisiones distintas. Algunos archivos pueden no tener estilos todavía, y si los tienen, probablemente fue con CSS inline (dentro del propio HTML con el atributo style="..."):

<h1 style="color: blue; font-size: 32px;">Bienvenido</h1>
<p style="color: gray; font-style: italic;">Iniciá sesión para continuar</p>

El CSS inline anda: la página se ve, los colores aparecen. Pero no escala. Si querés que ese mismo título tenga el mismo color en cinco archivos, lo escribís cinco veces; cuando lo cambies, lo cambiás cinco veces. Y el HTML termina lleno de detalles visuales que distraen de la estructura del contenido.

La forma profesional: archivo aparte

Todos los estilos viven en un único archivo css/estilos.css, linkeado una sola vez desde cabecera.php. Como cabecera.php se incluye en todas las páginas con include, todas las páginas heredan el link y aplican los mismos estilos automáticamente.

1) En cabecera.php, dentro del <head>, agregás el <link>:

cabecera.php (fragmento)
------------------------
<!DOCTYPE html>
<html lang="es">
<head>
  <meta charset="UTF-8">
  <title>Sistema</title>
  <link rel="stylesheet" href="css/estilos.css">
</head>
<body>
  <nav>
    <a href="index.php">Inicio</a>
    <a href="info.php?seccion=ayuda">Ayuda</a>
    <a href="info.php?seccion=contacto">Contacto</a>
    <a href="logout.php">Salir</a>
  </nav>

2) En css/estilos.css, escribís las reglas con selectores:

}

3) En el HTML quitás los style="..." y dejás los elementos limpios:

<h1>Bienvenido</h1>
<p>Iniciá sesión para continuar</p>

El estilo de h1 ya está en estilos.css y se aplica automáticamente porque la regla h1 { ... } apunta a todos los <h1> de la página. Eso es un selector de tipo: selecciona todos los elementos de ese tipo.

Antes y después

Antes (inline)Después (CSS aparte)
<h1 style="color: blue;">Hola</h1> HTML: <h1>Hola</h1>
CSS: h1 { color: blue; }
<nav style="background: blue;">...</nav> HTML: <nav>...</nav>
CSS: nav { background: blue; }
Regla DRY otra vez
Es la misma idea que usamos en clase 1 con cabecera.php y pie.php: lo común vive una sola vez. Acá lo común son los estilos.

Tipos básicos de selectores

SelectorQué seleccionaEjemplo
Tipo Todos los elementos de ese tipo HTML h1 { color: blue; }
Clase (.) Elementos con un class="..." específica .error { color: red; } aplica a <p class="error">
ID (#) Un único elemento con id="..." #login-box { ... } aplica a <div id="login-box">
El selector de clase es el que más vas a usar

Las clases (.) son la forma estándar de aplicar estilos en CSS profesional. Los selectores de tipo (h1, p) sirven para los estilos base que valen para todo el sitio (todas las páginas, todos los h1). Los IDs (#) prácticamente no se usan en la práctica para estilos: están reservados para identificar elementos únicos para JavaScript o anchors.

Las clases son lo que te permite tener variaciones: dos botones distintos, tres tipos de mensajes, cuatro variantes de tarjetas, sin tener que tocar la regla base de button o div.

Clases en la práctica: ejemplos típicos

1) Mensajes según tipo

Imaginá que en distintas páginas tu sistema muestra mensajes al usuario: avisos de éxito, advertencias, errores. En vez de inventar el estilo cada vez, definís tres clases y las reutilizás:

/* En estilos.css */
.mensaje {
  padding: 12px 16px;
  border-radius: 6px;
  margin: 12px 0;
  font-weight: 600;
}
.mensaje.exito        { background: #e6f4ea; color: #1e6f3a; }
.mensaje.advertencia  { background: #fff4e0; color: #8a5400; }
.mensaje.error        { background: #fde6e6; color: #9b1b1b; }
<!-- En cualquier .php -->
<p class="mensaje exito">Tu inscripción fue registrada.</p>
<p class="mensaje advertencia">Revisá que el DNI esté completo.</p>
<p class="mensaje error">Usuario o contraseña incorrectos.</p>

Cada <p> tiene dos clases separadas por espacio: mensaje (la base, común a los tres) y una más específica que define color (exito, advertencia o error). Eso se llama combinar clases.

2) Botones con variantes

/* En estilos.css */
.boton {
  padding: 10px 20px;
  border: none;
  border-radius: 4px;
  font-size: 15px;
  font-weight: 600;
  cursor: pointer;
}
.boton.primario   { background: #1e4fc2; color: white; }
.boton.secundario { background: #e1e3e0; color: #14161a; }

<button class="boton primario">Iniciar sesión</button>
<button class="boton secundario">Cancelar</button>
<button class="boton peligro">Cerrar sesión</button>

3) Tarjetas reutilizables

/* En estilos.css */
.tarjeta {
  background: white;
  border: 1px solid #e1e3e0;
  border-radius: 8px;
  padding: 20px;
  margin: 16px 0;
  box-shadow: 0 2px 6px rgba(0, 0, 0, 0.05);
}

  margin-top: 0;
  color: #1e4fc2;
}
<div class="tarjeta">
  <h3>Mi proyecto anual</h3>
  <p>Sistema de inscripciones para el kiosco escolar.</p>
</div>

<div class="tarjeta">
  <h3>Otro proyecto</h3>
  <p>Gestor de turnos para la salita.</p>
</div>

Mirá lo que pasó: definiste .tarjeta una sola vez en el CSS y la usaste dos veces (podrían ser veinte) en el HTML. Eso es DRY funcionando.

Convenciones para nombrar clases
  • En minúscula, con palabras separadas por guion (-): .boton-primario, .tarjeta-grande, .mensaje-error.
  • Que describan el rol, no el aspecto. Mejor .error que .rojo — porque si después cambiás el color del error, el nombre .rojo se vuelve mentira.
  • Cortas pero claras. Ni .b (no se entiende), ni .elBotonRojoDelLoginPrincipal (innecesariamente larga).
Regla mental para decidir entre tipo y clase
  • ¿Es algo que vale para todos los elementos de ese tipo en tu sitio? → selector de tipo (body, h1, form, nav).
  • ¿Es algo que aplicás a algunos elementos, según el rol que tienen? → selector de clase (.boton, .tarjeta, .error).

Cuidado con las rutas

Si el CSS no carga, casi siempre es por la ruta. La ruta del <link rel="stylesheet" href="..."> es relativa a la página que el navegador está mostrando, no al archivo cabecera.php en sí.

Si todas tus páginas están en la raíz de clase02/, href="css/estilos.css" funciona. Si tenés páginas en subcarpetas, tenés que ajustar.

Cómo verificar que el CSS está cargando
  1. Abrí la página en el navegador.
  2. Apretá F12 para abrir las DevTools.
  3. Andá a la pestaña Network (o Red).
  4. Refrescá la página (F5).
  5. Buscá el archivo estilos.css en la lista. Si dice 200, cargó bien. Si dice 404, la ruta está mal.
Truco: ruta absoluta desde la raíz del servidor

Si te complicás con rutas relativas, podés usar una ruta absoluta tipo href="/clase02/css/estilos.css". La barra del principio significa "desde la raíz del servidor" y funciona desde cualquier subcarpeta.

✓ Pausa para pensar

¿Por qué movemos los estilos a un .css aparte en lugar de dejarlos inline en cada .php?

Lo mismo aplica para JavaScript

Así como el CSS va en un archivo aparte, el código JavaScript también va en su propio archivo (por ejemplo js/scripts.js) y se linkea con <script src="js/scripts.js"></script> al final del <body>. Misma idea, misma regla DRY.

En esta clase no vamos a escribir JS, pero desde ahora la carpeta js/ ya forma parte de la estructura del proyecto.

3. Redirecciones server-side

Qué es un header HTTP

Cuando el servidor responde a una petición, antes del contenido (el HTML, la imagen, etc.) manda una lista de headers: información sobre la respuesta. Por ejemplo: qué tipo de contenido viene, qué encoding tiene, si hay cookies, cuánto tiempo cachearlo... Los headers son metadatos, no son el contenido en sí.

Ejemplos de headers HTTP típicos en una respuesta:

HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 1452
Set-Cookie: PHPSESSID=abc123; HttpOnly
Cache-Control: no-cache

<!DOCTYPE html>
<html>
... el HTML del body ...

Las líneas de arriba (antes del <!DOCTYPE html>) son los headers. El servidor las manda automáticamente; el navegador las lee y actúa en consecuencia.

El header Location: redirigir al navegador

PHP tiene una función llamada header() que sirve para agregar headers a la respuesta. El header que más vamos a usar este año se llama Location, y cuando lo mandamos le decimos al navegador "andate a otra URL":

header('Location: bienvenido.php');
exit;

Esto manda como respuesta:

HTTP/1.1 302 Found
Location: bienvenido.php

(sin body HTML)

El navegador, al recibir un 302 con un Location, automáticamente hace una nueva petición a esa URL. El usuario nunca ve un "HTML intermedio": ve directamente la página destino. La transición es transparente.

El esquema del redirect (status 302)

Navegador                              Servidor
   |                                       |
   |---- POST /procesar-login.php -------->|
   |     usuario=admin, clave=1234         |
   |                                       |  valida
   |                                       |  $_SESSION['usuario'] = 'admin'
   |                                       |
   |<---- 302 Found ---------------------- |
   |     Location: bienvenido.php          |
   |     (sin body HTML)                   |
   |                                       |
   |  ← el navegador entiende y pide       |
   |     automáticamente la nueva URL      |
   |                                       |
   |---- GET /bienvenido.php ------------->|
   |                                       |  verifica $_SESSION
   |<---- 200 OK ------------------------- |
   |     HTML: "Hola, admin"               |
   |                                       |

Lo interesante: el navegador hace dos peticiones aunque el usuario solo apretó "Entrar" una vez. La primera POST devuelve un 302, y al recibirlo el navegador automáticamente lanza una nueva petición GET a la URL del Location. Esa segunda página es la que el usuario termina viendo.

Tres reglas innegociables con header()
  1. header() tiene que ir antes de cualquier echo o HTML. Si ya escupiste una línea de HTML, después no podés mandar headers (te tira el famoso error "Headers already sent").
  2. Siempre exit; después del header('Location:'). Si no, PHP sigue ejecutando código después y se manda problemas.
  3. La ruta del Location es relativa al sitio, no al archivo PHP que la ejecuta.
¿Cuándo usar header('Location:')?
  • Después de procesar un formulario exitoso (login OK → ir a bienvenido).
  • Cuando una página interna recibe un usuario no logueado (echarlo al login).
  • Después de un logout (limpiar sesión → volver al login).
  • Cuando una página recibe datos inválidos y querés mandar al usuario "atrás" a corregir.

4. Navegación con links HTML

Hay dos formas de moverse entre páginas del proyecto. Las usamos ambas:

FormaQuién decideCuándo se usa
Link HTML
<a href="...">		 El usuario (clickea) Navbar, botones, menúes. Navegación visible.
Redirección server-side
header('Location:')		 El servidor (PHP) Después de procesar algo: validar login, redirigir tras logout, echar a un usuario no autorizado.

El link HTML es navegación que el usuario inicia. La redirección server-side es navegación que el servidor decide por el usuario.

5. Sesiones: la idea

Pensá en un servidor que recibe pedidos de mucha gente al mismo tiempo. Vos te logueás, pero la página siguiente que pidas no sabe que sos vos: HTTP es un protocolo sin memoria. Cada pedido es independiente. Para el servidor, sin sesiones, cada vez que pedís una página sos un desconocido nuevo.

Una sesión resuelve eso. Es una caja de datos que el servidor recuerda mientras vos tenés el navegador abierto. Cada usuario tiene su propia caja. El servidor sabe cuál caja es tuya por una llavecita (cookie) que el navegador le manda en cada pedido.

Cómo funciona por dentro

NAVEGADOR (del usuario)               SERVIDOR
                                       ┌──────────────────────────┐
   cookie:                             │  Cajas de sesión          │
   PHPSESSID = abc123       ──────►   │                          │
                                       │  abc123 → {usuario:admin}│
                                       │  xyz789 → {usuario:prof} │
                                       │  ...                     │
                                       └──────────────────────────┘
                                       (cada caja es un usuario distinto)

En cada petición:
  - El navegador manda su cookie PHPSESSID en los headers.
  - El servidor usa ese ID para encontrar la caja correcta.
  - El servidor lee/escribe $_SESSION en esa caja.
  - El usuario nunca ve el contenido de la caja, solo tiene la llavecita.

La cookie PHPSESSID es la llave. El contenido real ($_SESSION) vive en el servidor, no en la cookie. Por eso es seguro guardar ahí cosas sensibles como "este usuario está logueado": el cliente no puede falsificar el contenido, solo la llave.

PreguntaRespuesta
¿Dónde vive la caja?En el servidor.
¿Quién accede a mi caja?Sólo vos (mientras tu navegador esté abierto).
¿Cuánto dura?Hasta que cerrás el navegador o pasa un rato sin actividad.
¿Sirve para guardar datos en serio?No. Para eso usamos base de datos (más adelante).
¿Se pierde si cerrás el navegador?Sí. Eso es lo que diferencia una sesión de una BD.
¿Cuándo usar una sesión?
  • Cuando necesitás recordar quién es el usuario entre páginas (caso del login).
  • Cuando querés contar algo durante la visita (cuántas veces clickeó X, cuántos formularios completó).
  • Cuando un dato tiene que viajar de una página a otra sin pasarlo por URL ni por formulario.

6. Sesiones: cómo se usan

En PHP, una sesión es básicamente una variable especial llamada $_SESSION, que se comporta como un array asociativo.

<?php
  session_start();                  // abrir/recuperar la sesion

  $_SESSION['usuario'] = 'admin';   // guardar
  echo $_SESSION['usuario'];        // leer
  unset($_SESSION['usuario']);      // borrar una clave
  session_destroy();                // destruir toda la sesion
Las 3 reglas innegociables
  1. Siempre session_start(); al principio de cualquier archivo que use $_SESSION. Tiene que ir antes de cualquier echo o HTML (igual que header()).
  2. Verificar antes de usar: usar isset($_SESSION['clave']) antes de leer $_SESSION['clave'], sino PHP tira warning si la clave no existe.
  3. Para cerrar sesión: session_destroy() y después redirigir al login. No quedarse en la misma página con la sesión destruida.

Ejemplo aparte del login: contador de visitas

El uso más conocido de $_SESSION es el login, pero no es el único. Sirve para cualquier cosa que tenga que recordarse entre páginas del mismo usuario. Veamos un ejemplo simple: contar cuántas veces el usuario recargó una página durante su sesión actual del navegador.

contador.php
============
<?php
  session_start();

  // Si la clave 'visitas' todavia no existe, la inicializamos en 0
  if (!isset($_SESSION['visitas'])) {
    $_SESSION['visitas'] = 0;
  }

  // Aumentamos en 1 cada vez que se carga la pagina
  $_SESSION['visitas']++;
?>
<!DOCTYPE html>
<html>
<body>
  <h1>Visitaste esta página <?= $_SESSION['visitas'] ?> veces</h1>
</body>
</html>
Probalo y mirá cómo se comporta
  • La primera vez que entrás: muestra "Visitaste esta página 1 veces".
  • Apretás F5 (recargar): muestra 2. Y 3. Y 4.
  • Abrís una pestaña nueva y entrás de nuevo: sigue contando desde donde quedó (la sesión es la misma, son la misma "caja").
  • Cerrás el navegador entero, lo volvés a abrir y entrás: vuelve a 1. La sesión anterior se perdió.
  • Otro usuario en otra PC entra: tiene su propio contador, empieza en 1. Las sesiones no se comparten entre usuarios.
Tres cosas que ilustra este ejemplo
  1. Patrón "inicializar si no existe": el if (!isset(...)) evita el warning de la regla 2. Antes de usar una clave que tal vez no exista todavía, la creás con un valor por defecto.
  2. El valor de sesión es como cualquier variable: podés sumarle, restarle, concatenar, lo que necesites. $_SESSION['visitas']++ es lo mismo que $_SESSION['visitas'] = $_SESSION['visitas'] + 1.
  3. La sesión es por usuario y por navegador: dos personas, dos contadores; el mismo usuario en dos navegadores distintos, dos contadores; el mismo usuario en el mismo navegador con varias pestañas, un solo contador compartido.

7. Login con sesión: patrón completo

Este es el flujo del login completo. Tres archivos: el form, el procesador, la página interna. Más logout.

El flujo en un diagrama

index.php  --POST-->  procesar-login.php
                      session_start + valida
                      $_SESSION['usuario'] = X
                      header(Location: bienvenido.php)

bienvenido.php  --> session_start
                    if (!isset $_SESSION) ir a index

logout.php  --> session_destroy + ir a index

Cada flecha es un viaje del usuario entre páginas. El POST es la única transición que dispara el usuario (apretando "Entrar" en el formulario). Las demás son redirects server-side (header('Location:')) o verificaciones.

index.php — el formulario de login

<?php include 'includes/cabecera.php'; ?>

<h1>Iniciar sesión</h1>

<form action="procesar-login.php" method="POST">
  <label>Usuario:
    <input type="text" name="usuario">
  </label>
  <label>Contraseña:
    <input type="password" name="clave">
  </label>
  <button type="submit">Entrar</button>
</form>

<?php include 'includes/pie.php'; ?>

procesar-login.php — valida y abre sesión

<?php
  session_start();

  $usuario = $_POST['usuario'] ?? '';
  $clave   = $_POST['clave']   ?? '';

  if ($usuario == 'admin' && $clave == '1234') {
    $_SESSION['usuario'] = $usuario;
    header('Location: bienvenido.php');
    exit;
  } else {
    header('Location: index.php');
    exit;
  }
Línea por línea
  • session_start(): abre o recupera la sesión de este usuario.
  • $_POST['usuario'] ?? '': agarra el dato del form, con default vacío si no vino.
  • if ($usuario == 'admin' && $clave == '1234'): valida contra usuario y contraseña hardcodeados. El && es "y lógico": las dos condiciones tienen que cumplirse.
  • $_SESSION['usuario'] = $usuario: guarda el nombre del usuario en su caja de sesión.
  • header('Location: bienvenido.php') + exit: redirige a la página interna.
  • Si la validación falla: redirige de vuelta al login.

logout.php — cierra la sesión

<?php
  session_start();
  session_destroy();
  header('Location: index.php');
  exit;
Aviso importante sobre la contraseña en duro
La validación $usuario == 'admin' && $clave == '1234' nunca se usa así en producción. La contraseña real se guarda hasheada en base de datos y se compara con password_verify(). Hoy escribimos así porque no tenemos base de datos y la idea es entender el mecanismo de sesión, no el de seguridad. La autenticación real la vemos cuando entremos a MySQL.

8. Protegiendo páginas internas

Tener un login que abre sesión no alcanza. Cada página interna tiene que verificar que el usuario esté logueado, sino cualquiera puede entrar escribiendo la URL directamente.

Los dos escenarios posibles

CASO A — Usuario logueado entra a bienvenido.php
─────────────────────────────────────────────────
Navegador                          Servidor (bienvenido.php)
   |                                   |
   |---- GET /bienvenido.php --------->|
   |     cookie: PHPSESSID=abc123      |  session_start()
   |                                   |  $_SESSION['usuario'] = 'admin' ✓
   |                                   |  isset() → true → CONTINÚA
   |<---- 200 OK ----------------------|
   |     HTML "Hola, admin"            |


CASO B — Alguien escribe la URL directa SIN loguearse
─────────────────────────────────────────────────────
Navegador                          Servidor (bienvenido.php)
   |                                   |
   |---- GET /bienvenido.php --------->|
   |     (sin cookie o cookie vacía)   |  session_start()
   |                                   |  $_SESSION['usuario'] no existe
   |                                   |  isset() → false → REDIRIGE
   |<---- 302 ------------------------ |
   |     Location: index.php           |
   |                                   |
   |---- GET /index.php -------------->|  ← termina en el login
   |<---- 200 OK (formulario login) -- |

Las 4 líneas del patrón de protección hacen exactamente esa decisión: si tenés sesión, te dejo entrar; si no, te mando al login. Sin esas líneas, la página es pública aunque exista un login en el sitio.

bienvenido.php (protegido por sesión)
-------------------------------------
<?php
  session_start();

  if (!isset($_SESSION['usuario'])) {
    header('Location: index.php');
    exit;
  }
?>
<?php include 'includes/cabecera.php'; ?>

<h1>Hola, <?= $_SESSION['usuario'] ?></h1>

<?php include 'includes/pie.php'; ?>

Esas 4 líneas de session_start + if !isset son el patrón de protección. Van al principio de toda página interna del sistema. Si no están, esa página es pública.

El "ajá" del login
Una vez que entendés esto, entendés el login completo: el form pide datos, el procesador valida y guarda en sesión, las páginas internas verifican la sesión antes de mostrar nada, el logout destruye la sesión. Esa es toda la mecánica de autenticación clásica en PHP.

9. GET aplicado: secciones dinámicas

En clase 1 vimos GET conceptualmente: los datos viajan en la URL después del ?. Hoy lo usamos para algo concreto: una sola página que muestra contenido distinto según un parámetro.

En la navbar (que vive en cabecera.php) hay tres links:

<a href="info.php?seccion=ayuda">Ayuda</a>
<a href="info.php?seccion=contacto">Contacto</a>
<a href="info.php?seccion=nosotros">Nosotros</a>

Los tres llevan a la misma página (info.php), pero con un parámetro distinto en la URL.

El flujo de GET en un diagrama

NAVBAR (links)               info.php
[Ayuda]    --GET-->          $_GET['seccion'] = 'ayuda'
[Contacto] --GET-->          $_GET['seccion'] = 'contacto'
[Nosotros] --GET-->          $_GET['seccion'] = 'nosotros'

                             el if/elseif decide qué bloque
                             de HTML mostrar segun el valor

Cada link es un GET distinto a la misma página. El parámetro ?seccion=... es el dato que el usuario "le pasa" a info.php para que decida qué mostrar. PHP recibe ese valor en $_GET['seccion'].

URL que clickea el usuarioValor que llega a PHPBloque que se muestra
info.php?seccion=ayuda$_GET['seccion'] = 'ayuda'Bloque Ayuda
info.php?seccion=contacto$_GET['seccion'] = 'contacto'Bloque Contacto
info.php?seccion=nosotros$_GET['seccion'] = 'nosotros'Bloque Nosotros
info.php (sin parámetro)$_GET['seccion'] = nullBloque por defecto (gracias al ??)

info.php — patrón completo

<?php
  session_start();
  if (!isset($_SESSION['usuario'])) {
    header('Location: index.php');
    exit;
  }

  $seccion = $_GET['seccion'] ?? 'ayuda';
?>
<?php include 'includes/cabecera.php'; ?>

<?php if ($seccion == 'ayuda'): ?>
  <h1>Ayuda</h1>
  <p>Cómo usar el sistema...</p>
<?php elseif ($seccion == 'contacto'): ?>
  <h1>Contacto</h1>
  <p>Escribinos a contacto@ejemplo.com</p>
<?php elseif ($seccion == 'nosotros'): ?>
  <h1>Nosotros</h1>
  <p>Somos un equipo de 7° año...</p>
<?php endif; ?>

<?php include 'includes/pie.php'; ?>
Tres cosas para mirar acá
  1. info.php también está protegida por sesión: las 4 líneas iniciales son el mismo patrón que en bienvenido.php.
  2. $_GET['seccion'] ?? 'ayuda' agarra el valor de la URL, y si no hay nada usa 'ayuda' por defecto. Sin el default, PHP tiraría warning si entran a info.php sin ?seccion=....
  3. El if/elseif con la sintaxis <?php if (...): ?> ... <?php endif; ?> es sintaxis alternativa, equivalente a las llaves pero más legible cuando hay mucho HTML adentro. Las dos formas son válidas.
¿Por qué no tres archivos separados?
Podrías hacer ayuda.php, contacto.php, nosotros.php. Pero entonces tendrías que repetir el session_start, la protección, el include de cabecera y la estructura en cada archivo. Con un solo info.php parametrizado por GET, el código común vive una sola vez (DRY otra vez).

10. Errores frecuentes

"Headers already sent by..." (el error más típico de hoy)

Te aparece cuando ya escupiste algo (un echo, un espacio, una línea en blanco antes del <?php) y después intentás llamar a header() o session_start().

Soluciones:

  • Asegurate que el <?php sea la primerísima cosa del archivo. Sin espacios, sin líneas en blanco antes.
  • Si el archivo no tiene HTML, no cierres con ?>. Dejá el archivo abierto; PHP no se queja.
  • Revisá si tu editor está agregando un BOM al guardar (caracteres invisibles al principio del archivo). Notepad++ y VS Code permiten cambiarlo.
$_SESSION se "pierde" entre páginas

Te logueás, redirige a bienvenido.php, y $_SESSION['usuario'] está vacío.

Causa típica: falta session_start() en alguna de las páginas. Hay que ponerlo en todas las que usan $_SESSION, no solo en el procesador.

Olvidar exit; después de header('Location:')

PHP sigue ejecutando código después del header. Esto puede causar comportamiento raro: la redirección sí ocurre, pero antes se procesan cosas que no querías que se procesaran.

Regla mental: header y exit van siempre juntos. Como un signo de puntuación.

Login que valida pero no protege

El usuario se loguea bien, pero después puede entrar a bienvenido.php escribiendo la URL directamente sin pasar por el login.

Causa: falta el patrón de protección al inicio de bienvenido.php (las 4 líneas de session_start + if !isset + redirect). Sin eso, la página es pública aunque exista un login.

CSS no carga

Casi siempre es la ruta del <link>. Verificá en el navegador con F12 → Network → buscar el archivo CSS. Si dice 404, la ruta está mal.

$_GET['seccion'] tira warning

Si entrás a info.php sin el ?seccion=..., la clave no existe y PHP tira warning. Solución: usar el operador ?? con un default, igual que con $_POST: $_GET['seccion'] ?? 'ayuda'.

Vocabulario que se suma al de Clase 1:

PalabraDefinición
SesiónCaja de datos que el servidor recuerda para un usuario mientras tiene el navegador abierto.
$_SESSIONVariable especial de PHP donde se guardan/leen los datos de la sesión.
session_start()Abre la sesión (o la recupera si ya existe). Va al principio de cada archivo que use $_SESSION.
session_destroy()Destruye toda la sesión. Se usa en el logout.
Header HTTPInformación que el servidor manda al navegador antes del contenido. Location: es uno de los muchos headers posibles.

unset()Borra una variable o una clave específica de un array.

12. Buenas prácticas

  1. Siempre session_start() al principio, antes de cualquier HTML.
  2. Siempre exit; después de header('Location:'). Como un signo de puntuación: van juntos.
  3. Proteger todas las páginas internas con el patrón if (!isset($_SESSION['usuario'])). Si no, son públicas.
  4. Linkear el CSS una sola vez desde cabecera.php. Nunca repetir el <link> en cada archivo.
  5. Default en los parámetros GET: $_GET['x'] ?? 'algo'. Sino, warning cuando no llega.
  6. Cuando algo no anda, var_dump() tu sesión: var_dump($_SESSION); al principio del archivo te muestra qué tiene la caja. Es el debug más rápido.
  7. No escribir el ?> de cierre en archivos que sólo tienen PHP. Evita errores de "headers already sent" por espacios al final.
