2. Fundamentación

El 7.º año es el tramo final de la formación técnica. En este espacio, los estudiantes integran los conocimientos construidos durante toda la tecnicatura para diseñar, desarrollar y presentar un sistema informático real, trabajando en equipo y con una metodología similar a la del ámbito profesional.

La materia se organiza como un taller de proyecto anual: cada grupo identifica una necesidad concreta (de la escuela, del barrio, de una organización o de un usuario tipo) y la convierte en un sistema funcional. El recorrido no se centra en aprender herramientas aisladas, sino en aprender a decidir qué herramienta usar y por qué, documentar lo que se hace y poder explicarlo.

Se trabaja con tecnologías ampliamente utilizadas en el mercado y disponibles en cualquier equipo: HTML, CSS, Bootstrap, JavaScript, PHP, MySQL y Git/GitHub. A medida que el grupo avanza, se incorporan herramientas más potentes como Laravel o APIs, siempre como profundización sobre lo ya aprendido, no como un salto técnico forzado.

3. Objetivos del espacio curricular

Al finalizar el ciclo lectivo, se espera que los estudiantes sean capaces de:

  • Llevar adelante un proyecto informático completo, desde el relevamiento inicial hasta la presentación final.
  • Diseñar interfaces de usuario claras, accesibles y consistentes, aplicando criterios básicos de usabilidad.
  • Modelar datos en MySQL, escribir consultas SQL y vincular la base de datos con una aplicación PHP.
  • Desarrollar aplicaciones web con PHP, incorporando buenas prácticas de organización del código.
  • Utilizar Git y GitHub como herramienta cotidiana de trabajo colaborativo.
  • Documentar el proyecto de manera clara: requerimientos, decisiones de diseño, manual de uso.
  • Trabajar en equipo, distribuyendo tareas, respetando acuerdos y resolviendo conflictos técnicos.
  • Comunicar oralmente y por escrito el funcionamiento del sistema desarrollado.
  • Reconocer cuándo conviene incorporar herramientas más avanzadas (frameworks, APIs) y poder fundamentar la decisión.

4. Contenidos por etapas

El programa se organiza en las cuatro etapas del enfoque ABP. Cada etapa tiene contenidos técnicos y producciones esperadas.

Etapa 1 — Exploración y contextualización

Propósito: identificar el problema a resolver y el contexto en el que se va a usar el sistema.

  • Conformación de equipos de trabajo y acuerdos internos.
  • Identificación de un problema o necesidad real. Análisis de usuarios destinatarios.
  • Investigación de sistemas existentes que resuelvan problemas similares.
  • Repaso y nivelación de HTML y CSS. Estructura semántica de una página.
  • Introducción a Bootstrap: grilla, componentes básicos, diseño responsive.
  • Git y GitHub desde el primer día: creación de repositorio, commits, ramas básicas, trabajo colaborativo.

Producción de la etapa: documento de relevamiento (problema, usuarios, alcance), mockups o bocetos de las pantallas principales, repositorio creado en GitHub con un README inicial.

Etapa 2 — Análisis y diseño del sistema

Propósito: definir qué va a hacer el sistema, cómo van a moverse los datos y cómo va a verse.

  • Requerimientos funcionales y no funcionales. Casos de uso simples.
  • Diseño de la base de datos: entidades, atributos, relaciones. Diagrama entidad-relación.
  • MySQL: creación de tablas, claves primarias y foráneas, normalización básica.
  • SQL: SELECT, INSERT, UPDATE, DELETE, JOIN, WHERE, ORDER BY, GROUP BY.
  • Diseño de interfaces: wireframes, prototipos navegables. Criterios de usabilidad básica (jerarquía visual, contraste, claridad de los formularios).
  • Accesibilidad básica: etiquetas, contraste, navegación por teclado, textos alternativos.
  • JavaScript básico para validaciones visuales de formularios (campos requeridos, formato, feedback al usuario).

Producción de la etapa: diagrama entidad-relación, script SQL de creación de la base, prototipo de interfaz (puede ser en HTML+CSS+Bootstrap sin lógica), documento de requerimientos.

Etapa 3 — Desarrollo y programación

Propósito: construir el sistema. Es la etapa más larga del año.

  • Estructura de un proyecto PHP organizado: separación de vistas, lógica y acceso a datos.
  • Conexión PHP–MySQL. Consultas preparadas. Prevención de inyección SQL.
  • Implementación de CRUD completo sobre las entidades del proyecto.
  • Manejo de sesiones, login y control de acceso por rol.
  • Validación de datos en cliente (JavaScript) y en servidor (PHP). Por qué siempre se valida en el servidor.
  • Manejo de errores y mensajes al usuario.
  • Buenas prácticas: no guardar contraseñas en texto plano (uso de password_hash), no exponer datos sensibles, no confiar en lo que viene del navegador.
  • Trabajo continuo con Git: ramas por funcionalidad, pull requests internos del equipo, resolución de conflictos.
  • Introducción progresiva a Laravel (segunda mitad del año, según avance del grupo): qué es un framework, qué problemas resuelve, comparación con el PHP que ya escribieron. Rutas, controladores y vistas. Si el tiempo y el grupo lo permiten, migración parcial del proyecto a Laravel o desarrollo de un módulo nuevo dentro del framework.
  • Contenidos complementarios opcionales (solo si el grupo avanza con holgura): consumo de una API pública desde el proyecto, profundización en Laravel (middleware, Eloquent ORM, autenticación). Estos contenidos no son obligatorios ni condicionan la aprobación.

Producción de la etapa: sistema funcional desplegado en local con todas las funcionalidades núcleo, base de datos cargada con datos de prueba, repositorio actualizado.

Etapa 4 — Validación y presentación

Propósito: poner a prueba el sistema, corregir lo que haga falta y comunicarlo.

  • Pruebas funcionales con usuarios reales o simulados. Registro de errores y mejoras.
  • Corrección de bugs y ajustes finales. Trabajo con el repositorio para llegar a una versión estable.
  • Documentación final: manual de usuario, manual técnico breve, README del repositorio.
  • Preparación de la presentación oral: demo en vivo, explicación de decisiones técnicas, distribución de roles dentro del equipo.
  • Reflexión final: qué aprendieron, qué dificultades tuvieron, qué harían distinto.

Producción de la etapa: sistema en su versión final, documentación completa, presentación oral ante docentes y compañeros.

6. Modalidad de trabajo

  • Taller con producción permanente. Cada clase combina un momento corto de presentación de contenido y un bloque extenso de trabajo en el proyecto.
  • Equipos estables de 3 a 4 estudiantes durante todo el año, con roles rotativos (coordinación, frontend, backend, documentación).
  • Bitácora del grupo: cada equipo lleva un registro semanal de avances, problemas encontrados y decisiones tomadas. Forma parte de la evaluación.
  • Entregas parciales obligatorias al cierre de cada etapa. No se acumulan al final.
  • Trabajo continuo en GitHub: todo el código vive en el repositorio del equipo. El docente puede revisar el avance en cualquier momento.
  • Acompañamiento docente en el aula durante el desarrollo. Se prioriza que los estudiantes resuelvan, no que reciban la solución hecha.
  • Articulación con la otra materia de 7.º que también trabaja con PHP y MySQL: se busca evitar superposiciones y aprovechar los aprendizajes en ambos espacios.

7. Evaluación

La evaluación es continua, por etapas y por proceso. No hay un único examen final: la calificación surge del conjunto de producciones del año.

Instancia Qué se evalúa Peso aproximado
Etapa 1 — Relevamiento y bocetos Claridad del problema, calidad de los mockups, organización del repositorio 10 %
Etapa 2 — Diseño y base de datos Diagrama ER, script SQL, prototipo de interfaz, requerimientos 20 %
Etapa 3 — Desarrollo Sistema funcional, calidad del código, uso de Git, validaciones, seguridad básica 35 %
Etapa 4 — Versión final y presentación Estabilidad del sistema, documentación, defensa oral 25 %
Trabajo sostenido Bitácora, participación, cumplimiento de entregas, trabajo en equipo 10 %

Criterios generales que se aplican a todas las instancias:

  • Funcionamiento real de lo entregado, no solo de la apariencia.
  • Coherencia entre lo diseñado, lo documentado y lo construido.
  • Calidad del código: organización, nombres claros, ausencia de errores graves.
  • Uso correcto de la base de datos y del versionado.
  • Capacidad del equipo para explicar lo que hicieron y por qué.
  • Cumplimiento de los plazos pactados.

8. Instrumentos de evaluación

  • Rúbricas por etapa, entregadas al inicio de cada una, donde se explicita qué se va a observar.
  • Revisión del repositorio en GitHub: commits, ramas, historial de cambios, README.
  • Bitácora del equipo (digital o en papel), revisada periódicamente.
  • Demostraciones funcionales al cierre de cada etapa: el grupo muestra lo que tiene corriendo.
  • Defensa oral final: presentación del sistema, preguntas del docente y de compañeros.
  • Coevaluación entre pares dentro del equipo al cierre del año, para reflexionar sobre el aporte individual.
  • Autoevaluación escrita al finalizar cada etapa.

9. Bibliografía y recursos para estudiantes

Documentación oficial (referencia principal):

  • MDN Web Docs — HTML, CSS y JavaScript: developer.mozilla.org/es
  • PHP — Manual oficial en español: php.net/manual/es
  • MySQL — Documentación oficial: dev.mysql.com/doc
  • Bootstrap — Documentación: getbootstrap.com
  • Git — Libro Pro Git en español: git-scm.com/book/es/v2
  • Laravel — Documentación oficial: laravel.com/docs

Recursos de práctica recomendados:

  • W3Schools — repaso rápido de sintaxis (HTML, CSS, SQL, PHP).
  • SQLBolt — ejercicios interactivos de SQL.
  • Roadmap.sh — mapas de aprendizaje de desarrollo web (orientativo, no obligatorio).

Herramientas de trabajo:

  • Editor de código: Visual Studio Code (recomendado) o el editor que el estudiante prefiera.
  • Entorno PHP/MySQL local: XAMPP, Laragon o equivalente.
  • Cliente MySQL: phpMyAdmin o DBeaver.
  • Control de versiones: Git instalado localmente, cuenta personal en GitHub.
  • Diseño de mockups: Figma (versión gratuita) o lápiz y papel para los primeros bocetos.
